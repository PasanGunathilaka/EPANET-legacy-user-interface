# Codebase Report: EPANET Legacy UI

**Path analyzed:** . (repository root: C:\Learnings\Projects\EPANET-legacy-user-interface)
**Date analyzed:** 2026-07-21

## Tech Stack

- Borland/Delphi 7 Professional (Object Pascal), a Windows VCL desktop GUI application producing `epanet2w.exe` as an MDI (Multiple Document Interface) app. Confirmed by epanet2w/README.md: "The code has been compiled successfully with the Professional Version of Borland's Delphi 7," and by epanet2w/Epanet2w.dof's `[FileVersion]` section: "Version=7.0". epanet2w/Epanet2w.dpr is a standard Delphi program entry point ("Application.Initialize; Application.Title := 'EPANET 2'; Application.CreateForm(TMainForm, MainForm); Application.Run;").
- The compiler configuration file epanet2w/Epanet2w.cfg contains developer-machine-specific Delphi 7 install paths — `-LE"c:\program files\borland\delphi7\Projects\Bpl"` — reinforcing that this targets the classic (pre-.NET, pre-Unicode) Delphi 7 toolchain specifically, not a later Delphi version.
- Third-party charting is provided by TeeChart, confirmed by epanet2w/Fgraph.pas's uses clause ("TeEngine, Series, ExtCtrls, TeeProcs, Chart") and by components/ChartDlg.pas's own header comment describing itself as a dialog "of a TeeChart TChart."
- LOC-by-extension from the collected facts shows the codebase is essentially all Object Pascal + Delphi form definitions: `pas`=39615 lines and `PAS`=5773 lines (component units in components/ use the uppercase `.PAS` extension, e.g. TB97.PAS), plus `dfm`=1960 lines (the visual layout definitions for the project's 38 forms per epanet2w/README.md).
- No modern build system exists: no makefile, `.sln`, `.dproj`, or build script of any kind was found in the repository (confirmed by directory search of the whole tree) — the project is built solely by opening Epanet2w.dpr in the Delphi 7 IDE.

## Dependencies

- The one manifest detected, epanet2w/Epanet2w.dpr, is a Delphi *program* file, not a package manifest — its `uses` clause simply enumerates all ~60 in-repo units, e.g. `Fmain in 'Fmain.pas' {MainForm}`, confirming this is a single monolithic EXE with no externally-declared package dependencies in the modern sense.
- Real third-party/runtime package dependencies instead live in epanet2w/Epanet2w.dof: `Packages=TEE30;VCL30;VCLSMP30;QRPT30;TEEUI30;TEEDB30;VCLX30;VCLDB30;VCLDBX30` — i.e. Delphi's standard VCL runtime packages plus the TeeChart suite (TEE30/TEEUI30/TEEDB30) and QuickReport (QRPT30). Only the TeeChart packages could be confirmed as actually used in source (Fgraph.pas's `uses TeEngine, Series, ExtCtrls, TeeProcs, Chart`); no `QuickRpt`/`QRPrntr`/`Qrctrls` unit references were found anywhere in the `*.pas` tree, suggesting QRPT30 may be a stale/unused package reference left in the project options.
- The application's core computational dependency is `EPANET2.DLL`, bound via epanet2w/epanet2.pas's `external` declarations, e.g. `function ENepanet; external 'EPANET2.DLL';` and `function ENsolveH; external 'EPANET2.DLL';`. This DLL — the actual hydraulic/water-quality solver engine — is confirmed to be outside this repository; epanet2.pas is only the Pascal import/binding interface (its own header states: "Declarations of imported procedures from the EPANET PROGRAMMERs TOOLKIT (EPANET2.DLL)").
- Three further runtime dependencies are named but not included in the repo, per epanet2w/README.md: "works together with the following files that are all part of the EPANET 2 distribution package: epanet2.dll, epanet2.hlp, epanet2.cnt, and tutorial.hlp" — confirmed also by epanet2w/consts.txt's `HLPFILE = 'epanet2.hlp'; TUTORFILE = 'tutorial.hlp';` constants.
- Internally, epanet2w/ depends on components/ (a separate custom Delphi component library) per root README.md: "components.zip -- Custom Delphi components that must be installed into the Delphi IDE's component palette before working with the GUI code." This is confirmed directly in code: Fmain.pas's uses clause imports `PgSetup, OpenDlg, TB97, ChartDlg, Xprinter` — all units physically located in components/.
- Not all of components/ is documented: components/README.md lists exactly "7 custom components" (ChartDialog, OpenTxtFileDialog, NumEdit, PageSetupDialog, ToolBar97, VirtualListBox, PrintControl), but components/ComboColor.pas is also present and is credited to a different, third-party author in its own header comment: "Author: Fredrik Nordbakke - FNProgramvare (c) 1997" — this dependency's provenance/license is not tracked in the project's own README.

## Structure/Architecture

- Two top-level directories reflect a genuine two-library split, matching root README.md's description: components/ (reusable custom VCL controls, "last updated on 7/5/06") and epanet2w/ (the application itself, "last updated on 2/25/08").
- Inside epanet2w/, a consistent Hungarian-style naming convention separates three architectural layers, verified by opening representative files:
  - `Fxxxx.pas` = MDI parent/child window forms. Fmain.pas's own header states: "This unit contains the main MDI parent form, MainForm. It handles all user selections from the main menu." epanet2w/README.md confirms the remaining `F*` units (Fmap, Fbrowser, Fproped, Fovmap, Fsummary, Fsimul, Fstatus, Ftable, Fgraph, Fcontour, Fenergy, Fcalib) as MDI child or stay-on-top forms.
  - `Dxxxx.pas` = modal dialog forms (e.g. Dcurve.pas, Dcontrol.pas, Dsource.pas), each paired with a `.dfm` and enumerated with one-line purposes in epanet2w/README.md.
  - `Uxxxx.pas` = non-visual data/service units with no `.dfm` counterpart. Verified directly: Uglobals.pas ("Delphi Pascal unit that defines all global data types and constants used by EPANET2W"); Uinput.pas ("provides interface routines to EPANET2W's internal database" — its header comment doubles as a routine index: AddNode, AddLink, EditNode, EditLink, ValidateInput, etc.); Umap.pas (defines "the TMap object... contains drawing methods for rendering the Network Map on a memory bitmap"); Ufileio.pas ("reads network database from .NET file and saves network database to .NET file"); Uoutput.pas (retrieves simulation results via routines like GetNodeValue/GetLinkValue/GetNodeSeries/GetLinkSeries that read directly from a results file rather than calling the DLL again).
- State is centralized in a small number of large global singletons in Uglobals.pas rather than passed around explicitly: its top-level `var` block (lines 783-813) declares the entire in-memory network model (`Network : TNetwork; //Instance of Network object`) alongside UI/session flags (`FontName`, `BoldFonts`, `Blinking`, `FlyOvers`, `AutoBackup`, `HasChanged`, `UpdateFlag`, `CurrentList`, `CurrentItem`) — these are consumed directly across many other units (grep confirms references in components/PSForm.pas, epanet2w/Ddefault.pas, epanet2w/Dprefers.pas, epanet2w/Fmain.pas, epanet2w/Umap.pas).
- The solver-engine boundary is a two-phase file-mediated interop, verified by reading the actual call sequence: (1) epanet2w/Fsimul.pas exports the in-memory database to a temp `.INP` text file — `Uexport.ExportDataBase(TempInputFile,False);` — then drives the DLL solver purely through epanet2.pas's bindings: `err := ENopen(PChar(TempInputFile),PChar(TempReportFile), PChar(TempOutputFile));` followed by hydraulic/quality runs and `ENclose;`; (2) results are pulled back not via further DLL getter calls in epanet2w/Uoutput.pas but by directly reading a binary results file (its routines take a raw `var F: File` parameter, e.g. `procedure GetString(var F: File; var S: ShortString);`).
- Component reuse across the two folders is real and load-bearing, not incidental: Fmain.pas's uses clause `Printers, PgSetup, OpenDlg, TB97, ChartDlg, Xprinter, Uglobals, Uutils, ExtDlgs, ImgList` pulls directly from components/, meaning the two top-level folders cannot be built or versioned independently of each other.

## Domain

Inference: this is a desktop engineering/scientific tool for designing and analyzing municipal drinking-water pipe distribution networks, based on the domain vocabulary spanning the source (Uglobals.pas's object categories `JUNCS, RESERVS, TANKS, PIPES, PUMPS, VALVES` and property indices `ELEVATION, BASEDEMAND, PRESSURE, FLOW, VELOCITY, HEADLOSS`) and epanet2w/README.md's own framing: "EPANET is a pipe network analysis program that allows a user to create or import a pipe network database, compute the hydraulic and water quality behavior of the network, and selectively view simulation results in a variety of different formats."

Inference: water-quality modeling (chemical concentration, water age, source tracing) is a first-class feature alongside hydraulics, not an afterthought — evidenced by Uglobals.pas's `TWaterQuality = (wqNone, wqChem, wqTrace, wqAge)` type and epanet2.pas's water-quality constants (`EN_CHEM = 1; EN_AGE = 2; EN_TRACE = 3;`).

Inference: the product supports a model-calibration workflow comparing simulated results against field-measured data, evidenced by the dedicated CalibForm ("An MDI child form that compares simulated results against a set of measured values in tabular and graphical formats" per epanet2w/README.md, implemented in Fcalib.pas) plus the Dcalib1.pas/Dcalib2.pas dialogs for registering and configuring calibration data files.

Inference (low confidence): the intended end users are practicing engineers/utility staff rather than developers — inferred only from end-user help-file references in code (epanet2w/consts.txt's `HLPFILE = 'epanet2.hlp'; TUTORFILE = 'tutorial.hlp';`) and a full print/preview and dockable-toolbar UI (components/Xprinter.pas, components/TB97.PAS); no direct evidence of the actual user base (e.g. marketing copy, user manual) was read during this analysis.

## Risks/Tech-Debt

- No automated test suite exists anywhere in the repository (`test_locations` is empty, and no test-shaped files, directories, or DUnit references were found while reading source). For a circa-2008 Delphi 7 desktop app this is expected/typical rather than a newly discovered gap, but it does mean any refactor of the global-state-heavy units (Uglobals.pas, Uinput.pas, Umap.pas) has no regression safety net.
- Broad, information-losing exception handling in file I/O: epanet2w/Ufileio.pas repeatedly catches all exceptions without inspecting or logging the original error. For example, around lines 154-160: `FileStream := TFileStream.Create(Fname, fmCreate or fmOpenWrite or fmShareDenyWrite); except MessageDlg(FILE_ERR3a + Fname + '.' + #10 + #10 + FILE_ERR2b, mtError, [mbOK], 0); Exit; end;`, and around lines 421-426: `try FileStream := TFileStream.Create(Fname, fmOpenRead); except Result := FILE_NO_OPEN; Exit; end;`. In both cases the real exception (disk full, permission denied, path not found, file locked, etc.) is discarded in favor of one generic message, making field diagnosis of I/O failures difficult. Across all of epanet2w/*.pas, only one occurrence of typed exception handling (`on E: Exception do`, in Fsimul.pas line 161) was found; every other `except` block is untyped.
- Pervasive global mutable state: Uglobals.pas declares dozens of unit-level `var` globals directly under `interface`/implementation scope — including the entire in-memory network model (`Network : TNetwork;`), UI preference flags (`BoldFonts`, `Blinking`, `FlyOvers`, `AutoBackup`), and edit-session state (`HasChanged`, `UpdateFlag`, `CurrentList`, `CurrentItem`) — consumed directly by many other units (confirmed via grep hits in components/PSForm.pas, epanet2w/Ddefault.pas, epanet2w/Dprefers.pas, epanet2w/Fmain.pas, epanet2w/Umap.pas). This tight coupling makes isolated testing or incremental modernization of any single unit difficult without touching most of the codebase at once.
- Hardcoded, machine-specific build configuration is checked into source control: epanet2w/Epanet2w.cfg contains a specific original developer's filesystem paths — `-LE"c:\program files\borland\delphi7\Projects\Bpl"` and `-LN"c:\program files\borland\delphi7\Projects\Bpl"` — meaning anyone building this today must first edit this file to match their own Delphi 7 install location.
- Documentation/code drift in the component library: components/README.md documents exactly "7 custom components" (ChartDialog, OpenTxtFileDialog, NumEdit, PageSetupDialog, ToolBar97, VirtualListBox, PrintControl), but components/ also contains ComboColor.pas, a third-party control with a different, uncredited-in-README author (its own header states "Author: Fredrik Nordbakke - FNProgramvare (c) 1997") — meaning at least one dependency's provenance isn't captured in the project's own documentation, and the README cannot be trusted alone as a complete inventory of what's in components/.
- Toolchain fragility / single point of failure: no makefile, `.sln`, `.dproj`, or build script of any kind exists in the repo; the only documented way to build the software is to manually install Delphi 7 Professional plus every component in components/ onto the IDE's palette, per epanet2w/README.md's explicit warning: "Before loading the EPANET2W project into the Delphi IDE you must install the collection of EPANET2W custom components on to Delphi's Component Pallette." Anyone without a legacy Delphi 7 license/install cannot build this project at all.

## Suggested First Changes

- Start with epanet2w/Uglobals.pas before touching any form unit — it is imported, directly or transitively, by nearly every other unit examined (confirmed via the `uses` clauses of Fmain.pas, Ufileio.pas, Umap.pas, and Uinput.pas), and it is where the domain's core types (`TNetwork`, `TUnitSystem`, `TWaterQuality`) and the shared global mutable state are declared; understanding it first makes the rest of the codebase legible.
- If the goal is simply to get the project building, first fix epanet2w/Epanet2w.cfg's hardcoded `c:\program files\borland\delphi7\Projects\Bpl` search-library paths to the local Delphi 7 install location, then follow epanet2w/README.md's explicit prerequisite of installing every component in components/ onto the IDE's palette before opening Epanet2w.dpr.
- For a hydraulics/water-quality-results bug, the natural entry point is the pairing of epanet2w/Fsimul.pas (drives the solver via the `ENopen` / `ENsolveH` / `ENsolveQ` / `ENclose` sequence, confirmed around lines 141-167) and epanet2w/Uoutput.pas (reads the results back out via its own binary-file-reading routines such as `GetNodeValue`/`GetLinkValue`) — together these two files are the entire boundary between the GUI and the EPANET2.DLL solver engine.
- For a network-editing/data-model bug (e.g., adding, deleting, or validating a junction or pipe), start at epanet2w/Uinput.pas — its own header comment doubles as a routine index (`AddNode`, `EditLink`, `ValidateInput`, `ValidPipe`, etc.), making it possible to jump straight to the relevant routine in this 2243-line file without reading it end to end.
- Before modifying or relying on any custom component, cross-check components/README.md against the actual components/ directory listing first — as found during this analysis the two are out of sync (ComboColor.pas is present in the folder but absent from the README), so the README alone cannot be trusted as a complete component inventory.
